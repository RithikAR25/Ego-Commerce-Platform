# EGO Platform — End-to-End Test Plan

> **Version:** 2.1  
> **Date:** 2026-06-05  
> **Category Architecture:** 3-level enterprise hierarchy (ROOT → GROUP → LEAF) — migrated June 4, 2026.  
> **Show Password & Logo Micro-Interaction:** Added June 5, 2026 (Login + Register pages).  
> **Source:** Generated by full static analysis of all backend Java source, frontend TSX/TS, security config, DTOs, repositories, and service logic. Documentation was used only as supporting context — source code is the sole source of truth.  
> **Backend Base URL:** `http://localhost:8080`  
> **Frontend Base URL:** `http://localhost:5173`  
> **API Docs (Swagger UI):** `http://localhost:8080/docs`

> [!IMPORTANT]
> **Breaking change from v1.0:** `categoryId` / `categoryIds` search params have been removed. Use `categorySlug` (a URL-friendly slug string). Category tree API now returns `groups[]` (not `subcategories[]`) and each group carries `leafCategories[]`.

---

## How to Use This Document

Each test case follows this template:

| Field | Description |
|---|---|
| **Test ID** | Unique identifier (e.g. `AUTH-001`) |
| **Preconditions** | State required before the test starts |
| **Steps** | Numbered actions to perform |
| **Expected Result** | What success looks like |
| **DB Verification** | SQL or ORM queries to verify database state |
| **API Verification** | cURL/Swagger request to verify via API |
| **UI Verification** | What to observe in the browser |
| **Pass / Fail** | Tester marks result here |

> **Tip:** All API tests can be run via Swagger UI at `http://localhost:8080/docs`. Use the "Authorize" button to set `Bearer <token>` for authenticated requests.

## SECTION 1 — AUTHENTICATION

### AUTH-001 — Successful Registration
**Preconditions:** Clean user table (no account for test email).

**Steps:**
1. `POST /api/v1/auth/register` with body: `{ "email": "test@ego.com", "password": "Test@123", "firstName": "Test", "lastName": "User", "phone": "9876543210" }`
2. Record returned `accessToken` and `refreshToken`.

**Expected Result:**
- HTTP 201
- Body contains `data.accessToken`, `data.refreshToken`, `data.user.email`, `data.user.emailVerified: false`
- `data.user.role` = `"CUSTOMER"`

**DB Verification:**
```sql
SELECT id, email, is_email_verified, is_active, role, is_deleted
FROM users WHERE email = 'test@ego.com';
-- Expected: 1 row, is_email_verified=0, is_active=1, role=CUSTOMER, is_deleted=0
```

**API Verification:** Access token claims `sub` = email, `role` = `CUSTOMER`

**UI Verification:** Register form submits; user is redirected/logged in.

**Pass / Fail:** Pass

## MISSING COVERAGE

The following gaps were discovered in code that currently lack testability:

1. **Anonymous Cart (`cart:anon:<sessionId>`)**: The frontend uses `sessionId` from Zustand's `cartStore`. Test CART-008 requires manual Redis manipulation to seed an anonymous cart — no UI flow currently writes to `cart:anon:*`.

2. **`passwordChangedAt` JWT guard (AUTH-013)**: The `JwtAuthenticationFilter` checks `iat < user.passwordChangedAt`. Testing requires reducing token TTL or time-advancing DB timestamps — not possible in standard QA.

3. **Checkout lock race condition (CHK-006)**: True concurrent request testing requires load tools (k6, Gatling). Manual testing is impractical.

4. **ES synonym matching (SRCH-008)**: Requires ES to be running with the `product-index-settings.json` synonyms applied. If the index was created without synonyms, a reindex is needed first.

5. **Cart `/merge` endpoint**: The endpoint `POST /api/v1/cart/merge` is implemented in `CartService` but a dedicated CartController endpoint must be verified to exist. Anonymous merge is triggered from the frontend via Zustand store — not directly via API in the current auth flow.

6. **Razorpay live mode refund (RET-006 in live mode)**: Test mode simulates with `rfnd_TEST_*`. Live mode behavior requires a real `rzp_live_*` key — cannot be tested in dev.

7. **Email delivery verification**: Tests verify `notification_logs.status` but cannot confirm actual delivery to inbox without SendGrid activity feed access. `SENT` status is confirmed; `DELIVERED` requires webhook from SendGrid.

Discovered from source code analysis:

1. ~~**`CartController` for anonymous merge**~~: ✅ **RESOLVED** — `@PostMapping("/merge")` at `/api/v1/cart/merge` exists in `CartController.java` (line 137). Frontend `useAuth.ts` calls `mergeCart({ sessionId })` in both `useLogin` and `useRegister` `onSuccess` handlers. Fully wired end-to-end.

3. ~~**Resend verification rate-limiting**~~: ✅ **RESOLVED** — `RateLimitFilter.java` extended to cover `POST /api/v1/auth/resend-verification` with a limit of **3 requests per 60 minutes per IP** (more lenient than login/register since it is an authenticated endpoint). Bucket is keyed `IP + path` consistent with existing approach.

4. ~~**No per-user refresh token session limit**~~: ✅ **RESOLVED** — `RefreshTokenService.enforceSessionLimit()` implemented. On every new login/register (when `familyId == null`), active session count is queried. If `>= maxSessionsPerUser` (default 5, configurable via `ego.auth.max-sessions-per-user`), the **oldest active family is revoked** before the new token is issued. Uses two new JPQL queries in `RefreshTokenRepository`: `countActiveSessionsByUserId()` and `findActiveSessionsOldestFirst()`.

5. **`addressId` ownership at checkout**: `addressService.getForCheckout()` correctly validates ownership — test confirms the guard exists in `AddressService`.

6. **Shipping total always ZERO**: `buildOrder()` in `OrderService.java` hardcodes `shippingTotal = BigDecimal.ZERO`. No shipping calculation engine exists. Frontend should display "FREE" shipping clearly.

7. **Review moderation**: Reviews are auto-approved (`approved = 1` set immediately). No admin queue to reject reviews — test REV-006 (hard delete) is the only moderation action.
